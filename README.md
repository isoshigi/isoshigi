### いそしぎ忘備録

https://github.com/isoshigi/isoshigi-info-hub
https://blog.isoshigi.dev

検証結果や試行記録を物語として完成させ、小さな成果として蓄積していくアウトプット共有サイト。

個人開発における「完璧主義によるアウトプットの停滞」という課題に対し、意志力ではなく仕組みで解決する基盤構築に取り組みました。最高品質を求めるあまり不完全な成果物を公開できず、フィードバックや試行の機会を喪失していたことが課題でした。
そこで3つの施策を実行しました。1つ目は「行動マニフェスト」の策定で、失敗や中間成果物を「学んだ物語」として言語化し価値を持たせる運用規則を設けました。2つ目は、実践基盤としてブログを自作し、最小限の機能での段階的デプロイを敢行したことです。3つ目は、「価値を提供する者」という役割を定義し演じる心理的アプローチの導入です。
結果、ブログでの継続的なアウトプットが可能になりました。この経験から、自己の特性を否定せず仕組みで課題を迂回・解決するアプローチの有効性を学びました。

Tools: TypeScript, Astro, Cloudflare Workers Static Assets

### 有志で立ち上げた企業の基幹システム(構築中)

有志で立ち上げた企業の販売管理/在庫管理を中心としたシステム

有志の新規サービス開発プロジェクトにて、バックエンド兼インフラを1人で担当し、システム基盤構築に尽力しました。企画要件が頻繁に変わる中、セット販売などの複雑な仕様を、限られたリソースで保守性を保ちながら実装することが課題でした。
そこで3つの施策を実行しました。1つ目は、Go言語によるモジュラーモノリスの採用と、AIを活用したマッピング処理の自動化による開発効率の向上です。2つ目は、型安全性の担保と、ユーザー視点の「EARS形式」を用いたテスト設計への注力により、テスト実装工数を大幅に削減したことです。3つ目は、固定費を抑えるインフラ技術の選定です。
結果、1年間で44エンドポイント26テーブルを構築し、主要テストを網羅しました。この経験から、要件定義段階から技術と企画のトレードオフを合意する重要性を学び、現在はチームでDocs as Codeを推進しています。

Tools: Go, Gin, OpenAPI, Google Cloud (Cloud Storage, Cloud SQL, Cloud Run, and more...)

### FlowMatic

https://github.com/KaungSetLinn/FlowMatic

学校のチーム開発(二年次)で作成したプロジェクト管理グループウェア

学内の授業にて、5名チームのバックエンドリーダーとして直感的なプロジェクト管理ツール開発に尽力しました。目標は「経験の浅い学生」が迷わず使えるツールの完成でしたが、短い開発期間とメンバーの技術力不足が課題でした。
解決のため、3つの施策を実行しました。1つ目は、学習コスト低減のため既習のDjangoを採用し、検索のヒントを与え自己解決能力を養う指導でメンバーの技術力を底上げしたことです。2つ目は「2クリックで全機能に到達できる」というUXコンセプトの策定です。3つ目は、そのUXを実現しつつ開発速度を保つため「APIはシンプルに保つ」という方針をチームで事前合意し、フロントからの複数回API呼び出しを許容したことです。
結果、学内発表会でエキシビジョン枠に選出され、「説明不要で操作できる」と評価されました。この経験から、徹底したユーザー視点と、チーム開発における設計方針合意の重要性を学びました。

Tools: Python, Django Rest Framework, SQLite
